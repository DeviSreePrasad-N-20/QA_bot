# Document Q&A Bot — A RAG Pipeline with Citations

A command-line and web-based assistant that answers questions strictly from
your own PDF and Word documents, using Retrieval-Augmented Generation (RAG)
with Google Gemini and a local ChromaDB vector store. Every answer is
grounded in retrieved source text and cited by filename and page number.

## Why RAG?

LLMs have two structural limitations: they know nothing about your private
documents, and when pushed past what they actually know, they tend to
hallucinate plausible-sounding but incorrect answers. RAG sidesteps both
problems without any model fine-tuning: at query time, the user's question
is used to retrieve the most relevant chunks of your own documents, and
those chunks — not the model's training data — become the only source of
truth the model is allowed to answer from.

```
                                 +--------------------+
                                 |  Custom Documents  |
                                 | (PDF, DOCX, TXT)   |
                                 +---------+----------+
                                           |
                                           v
+-------------------+            +---------+----------+            +------------------+
|    User Query     +----------->|   Vector Database  +----------->| Retrieved Chunks |
+---------+---------+            |  (Semantic Search) |            +--------+---------+
          |                      +--------------------+                     |
          |                                                                 v
          |                      +--------------------+            +--------+---------+
          +--------------------->| Prompt Formulator  |<-----------+  Source Citations|
                                 +---------+----------+            +------------------+
                                           |
                                           v
                                 +---------+----------+
                                 |  Language Model    |
                                 |   (Gemini/LLM)     |
                                 +---------+----------+
                                           |
                                           v
                                 +---------+----------+
                                 | Grounded Answer &  |
                                 |    Citations       |
                                 +--------------------+
```

## Features

- Ingests PDF and DOCX files, tracking exact source filename and page number for every piece of text
- Recursive, paragraph-aware chunking with configurable overlap, so context isn't lost at cut boundaries
- Local, disk-persistent vector store (ChromaDB) — documents are embedded once and reused across sessions
- Strict grounding: the model is instructed to answer only from retrieved context, and to say so plainly when an answer isn't in the documents
- Inline citations in every answer, in `(filename, Page N)` format
- A relevance threshold that filters out retrieved-but-irrelevant chunks before they ever reach the LLM
- Works as both a CLI tool and a Streamlit web chat UI from the same entry point

## Setup

**1. Clone and enter the project**

```bash
git clone <your-repo-url>
cd document-qa-bot
```

**2. Create and activate a virtual environment**

```bash
python -m venv venv
source venv/bin/activate      # Windows: venv\Scripts\activate
```

**3. Install dependencies**

```bash
pip install -r requirements.txt
```

**4. Add your API key**

```bash
cp .env.example .env
# then edit .env and set GEMINI_API_KEY=your_key_here
```

Get a free key at [aistudio.google.com/app/apikey](https://aistudio.google.com/app/apikey).

## Usage

**Step 1 — Index your documents.** Drop any PDF or DOCX files into `data/`
(three sample documents are already included so you can try this
immediately), then run:

```bash
python -m src.ingest
```

Use `--rebuild` to wipe and re-index from scratch after changing chunk
settings or adding/removing source files:

```bash
python -m src.ingest --rebuild --chunk-size 800 --chunk-overlap 150
```

**Step 2 — Ask questions.**

Command line:
```bash
python -m src.main
```

Or the web UI:
```bash
streamlit run src/main.py
```

Example questions you can try against the included sample documents:
- "What was NorthBridge Analytics' net revenue in FY2026?"
- "What pricing tiers does NorthBridge Cloud Suite offer?"
- "What was the biomass yield improvement in the dynamic spectrum reactor?"
- "What is the company's marketing budget?" (not in the documents — the bot should say it can't find the answer, rather than guessing)

## Design Decisions & Analysis

**Chunking strategy.** Text is split recursively: first on paragraph breaks
(`\n\n`), then line breaks, then spaces, falling back to a hard character
cut only for the rare case of a single token longer than the chunk size
(e.g. a long URL). This keeps chunks aligned with natural semantic
boundaries instead of cutting mid-sentence. The default chunk size (1000
characters) balances two failure modes: chunks that are too small lose
surrounding context and produce noisy embeddings, while chunks that are too
large dilute the embedding's specificity and waste tokens at query time. A
200-character overlap between consecutive chunks ensures that information
sitting right on a cut boundary still appears intact in at least one chunk.

**Page tracking across formats.** PDFs have unambiguous page boundaries, so
page numbers are exact. DOCX files don't have fixed pagination without
rendering the document, so page numbers are approximated by detecting
explicit page-break runs in the document; a DOCX with no manual page breaks
is treated as a single logical page. This is called out explicitly so
citations are never silently wrong.

**Preventing hallucination.** Beyond the standard "use only the provided
context" system prompt, two additional safeguards are applied: (1) a
cosine-distance threshold (`MAX_DISTANCE_THRESHOLD` in `config.py`) drops
retrieved chunks that are too dissimilar from the query, even if they
technically ranked in the top-k, and (2) if every retrieved chunk is
filtered out this way, the pipeline returns the "I cannot find the answer"
message directly, without ever calling the LLM on irrelevant context. This
catches cases where a user asks about something genuinely absent from the
document set.

**Why ChromaDB over a hosted vector database.** For a project of this scope
(a handful of documents, single-user or small-team usage), a disk-persistent
local store avoids the operational overhead of running a separate database
service, while still supporting metadata filtering and configurable
distance metrics. The collection is created once during ingestion and
reused on every query without re-embedding.

**Separation of ingestion and querying.** `ingest.py` and `query.py` are
fully independent scripts that only share `config.py`. This means
re-indexing documents never requires touching query logic, and the query
path never accidentally triggers (costly, slow) re-embedding of the source
documents.

## Limitations & Possible Extensions

- DOCX page numbers are approximate for documents without manual page breaks (see above).
- Retrieval is purely semantic (vector similarity); there's no keyword/BM25 hybrid search, which can matter for queries containing exact terms, codes, or numbers.
- No support for scanned/image-only PDFs (would require OCR, e.g. `pytesseract`).
- The relevance threshold (`MAX_DISTANCE_THRESHOLD`) is a single global value; per-document or per-query-type thresholds could reduce false negatives/positives further.
- Conversation history isn't fed back into retrieval, so the bot can't currently resolve follow-up questions like "what about for FY2025?" without the user restating context.
